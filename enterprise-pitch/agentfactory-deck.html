The gap

Agents can write code. Enterprises can't govern them.

🌀 No persistent state

Chat-based agents forget everything between runs. No queue, no ownership, no history you can audit or resume.

🚫 No control gates

An agent that can merge, delete or approve its own work is a compliance non-starter for a regulated IT department.

❓ No delivery guarantee

"The agent said it's done" is not proof. Enterprises need PR-merged + pipeline-green as the definition of done.

AgentFactory separates the work (done by agents) from the control plane (owned by you) — so autonomy scales without giving up governance.

Why this exists
What it is

A persistent task board that is the state store & control plane for an external agent loop

  • The board never runs an agent itself. It owns lifecycle rules, state, and review.
  • Headless supervisors do the spawning. They start fresh claude -p / codex sessions that talk to the board over MCP.
  • A separate watcher verifies delivery over plain REST — PR merged & pipeline green.
  • Single source of truth. One local SQLite DB (WAL) holds every task, transition and comment.
Design principle

The agents do the work.
The board owns the truth.

Because state lives in the board — not in a conversation — every action is inspectable, resumable, and replayable. That property is what makes enterprise governance possible.

Supervisors (headless)

Polls queued, spawns one fresh agent session per task, releases crashed/timed-out claims.
reviewer
Polls in_review, spawns a session to post an advisory ai-review/v1 verdict.
watcher
Polls delivering, verifies PR/pipeline on GitHub / Azure DevOps over REST. No LLM, no spawn.

Every edge is declared

Transitions live in one table keyed by (from, to, by: human | agent). Any new state change goes there first; every op calls assertTransition.

Humans hold the gates

Approving a review, reopening, force-completing or pulling back a delivery are human-only edges. Agents can't delete or approve — by construction.

Delivery is machine-verified

The watcher owns delivering → done only when the PR is merged and checks are green — otherwise back to queued.

Lifecycle & guardrails
The transition · L1 → L5 autonomy

AgentFactory is how you operate at Levels 3–5 safely

1

Assisted

Human writes code; AI autocompletes. No autonomy.

2

Task copilot

AI handles a task under constant human steering, in the chat.

3

Bounded autonomy

Agent completes whole tasks inside guardrails; human approves each result.

4

Orchestrated fleet

Many tasks decomposed, queued & run in parallel; humans review by exception.

5

Self-delivering

Queue → build → review → merge → verified delivery with minimal human touch.

Autonomy is a property of the deployment

The same agent is safe or reckless depending on the controls around it. AgentFactory is those controls — so you can move up the levels without moving up the risk.

Where the gates sit defines your level

Loosen a human gate → move toward L5. Tighten it → drop to L3. The board makes that a configuration decision, task-by-task, not a rewrite.

Framing aligned to the Context& AI Transition model (learn.contextand.com/cs/ai-transition) and the widely-used 5-level agent autonomy scale.

The strategic centerpiece
Auditability & governance

Every decision is derived from an append-only log

  • Activity log is the source of truth. Metrics, analytics and review state are derived from it — so governance works retroactively, not just going forward.
  • AI review is walled off. Verdicts ride in ai-review/v1 comments; MCP payloads strip them so uncurated AI findings never reach the implementing agent.
  • Attribution on every action. Each transition records who — human or agent — and when.
  • Nothing is silently mutated. A version signal makes every change observable to the UI and to auditors.
What an auditor gets
  • Who queued the task
  • Which agent worked it, in which worktree/branch
  • What the human reviewer approved
  • Which PR delivered it & that CI passed

A complete, replayable chain of custody — per task.

Enterprise fit

Built to sit inside your existing controls

🔗 CI/CD-native

PR-based flow by convention: branch feature/<key>-<title> pushed to origin before results are submitted. The watcher confirms delivery on GitHub or Azure DevOps via REST — merged + green, or it bounces back to the queue.

🛡️ Security & data residency

Self-hosted. State is a local SQLite DB; services run as your own processes on your own network. Git host auth via GITHUB_TOKEN / AZDO_PAT env — no secrets in the board. Untrusted refs validated against SAFE_REF before touching git.

🧩 Isolation by design

Each task runs in its own git worktree under .worktrees/<key>. Fresh agent session per task — no shared, long-lived context to leak across work items.

Today it is single-user and local by design. The co-development work below is about taking these same guarantees to multi-user, SSO, and hardened deployment — without loosening any of them.
