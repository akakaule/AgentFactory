AgentFactory · system overview

Azure DevOps backlogs feed the board through per-workspace ado-bridge loops; pinned agents claim over MCP, work in isolated worktrees inside each workspace's git repo, and push feature/<key>-<title> branches back to origin. The board is the human checkpoint — and the only shared state is one SQLite file.

The loop, end to end

  1. 1Bridge claims a work item — unassigned (or #ai-agent-tagged) items in backlog order; the assignment PATCH is rev-guarded, so a concurrent human assignment wins.
  2. 2Design generation — claude -p turns the work item into a spec + acceptance criteria.
  3. 3Task created & queued — titled AB#123: … (the dedup key); the ADO item is tagged agentfactory · AF:<key>.
  4. 4Agent claims over MCP — get_next_task, FIFO within its pinned workspace; the payload carries spec, acceptance, prior feedback, pasted images, repoPath — and the protocol block: server-named branch + finish steps, computed fresh per claim.
  5. 5Isolated worktree — .worktrees/<key> on feature/<key>-<kebab-title>; parallel agents never collide.
  6. 6Finish protocol — commit → push -u origin → remove worktree. The pushed branch is the durable record.
  7. 7submit_result — guardrails verify pushed & clean (git reads only), then summary + branch link + token metrics → task lands In Review; unpushed work is refused.
  8. 8Human checkpoint — diff view on the board; Approve → done, or Request changes (feedback rides the next claim, same branch).
  9. 9Bridge closes the circle — posts the result summary to AB#123 and closes it. The human opens the PR from the pushed branch; CI fails → comment + Reopen ↩.

Rescue loops & invariants

↩Request changes — in_review → queued with feedback; the next claim continues on the same branch, so pushes update the same PR.
↩Reopen — done → queued when the PR build fails; paste the CI log as a comment first. Same branch, same PR.
⚑Release claim — a dead worker's in_progress task is re-queued by the human; full history survives for the next claimant.
◎One SQLite file is the only shared state — bridges (HTTP), agents (MCP), and the UI all converge on it; SSE fans changes out live. The board never runs agents and never writes to a repo.
⇄Scale by adding pairs — each new repo = one workspace + one pinned bridge + one pinned worker. Queues never leak across workspaces; analytics splits per worker/workspace.
