AgentFactory · how the agent loop works

A single-user local task board: humans create & review work in the web UI, an external agent (Claude Code or any MCP client) consumes and progresses it. The two sides never talk directly — a shared SQLite file is the only channel. Where tasks come from is the other half of the story: producers — from backlog to board and back.

    3 · The 6 MCP tools the agent gets

    ToolWhat it doesLoop role
    get_next_taskAtomically claim the oldest queued task (optionally scoped to a workspace) → in_progress; the payload includes a protocol block — server-named feature branch, worktree path, finish steps — the agent followsstart of every iteration
    get_taskFetch one task by key (spec, criteria, activity, links)read the full brief
    list_tasksList tasks, optionally filtered by statussituational awareness
    add_commentAppend a comment to the task's activity feednarrate progress (human sees it live via SSE)
    submit_resultVerify the finish protocol ran (branch pushed to origin, worktree removed), then store summary + links and move the task to in_reviewdeliver the work
    update_statusTransition the task (only agent-legal moves allowed)flag blocked / resume work

    Task creation is deliberately missing — only humans create work, via the web UI. The agent can never invent tasks for itself.

    4 · Wiring it up

    // terminal 1 — web UI (humans): npm run web:dev // agent side — MCP client config (.mcp.json in the target repo): { "mcpServers": { "agentfactory": { "command": "npx", "args": ["tsx", "c:\\Git\\AgentFactory\\packages\\mcp\\src\\index.ts"], "env": { "AGENTFACTORY_DB": "c:\\Git\\AgentFactory\\agentfactory.db" } } } }

    The MCP server isn't started by hand — the agent client launches it over stdio. The one rule: both sides must point AGENTFACTORY_DB at the same absolute path.
