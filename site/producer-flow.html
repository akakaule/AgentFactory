AgentFactory · producers: from backlog to board and back

Tasks don't have to be hand-written on the board. A producer is an agent that watches a source of work — here Azure DevOps, via ado-bridge — claims items, turns them into implementable specs with claude -p, queues them as AgentFactory tasks, and reflects every status change back to the source. The consumer half of the loop is covered in how the agent loop works.

Two independent loops meet at the board: the bridge produces queued tasks from the backlog; the worker consumes them. Each repo gets its own bridge, config, workspace and workspace-scoped PAT — parallel pairs never interfere.

2 · One work item, end to end

    3 · Marker-driven & self-healing

    🔑The title is the dedup key. Tasks are titled AB#123: …; before creating anything the bridge checks the board for that prefix. Re-running a pass can never duplicate work.
    🏷️Tag last, recover first. The ADO item gets agentfactory · AF:<key> as the final inbound step. An item claimed but not yet tagged (crash mid-pass) is re-selected by a recovery clause and resumed — no state file anywhere.
    🤝Rev-safe claims. The assignment PATCH carries a test op on the item's rev — if a human touched the item concurrently, the human wins and the bridge skips it.
    ♻️Idempotent outbound. The bridge only moves items whose state it set itself, and skips items already in the target state. Manually moved items are left alone; closing twice is a no-op.
    ⚙️Per-repo onboarding in one command. Run-AdoBridgeLoop -Init scaffolds the config, the worker wiring, the gitignore entries, and registers the workspace on the board.
    cd c:\git\your-repo Run-AdoBridgeLoop -Init # scaffold + register workspace Run-AdoBridgeLoop -StorePat # PAT → Credential Manager Run-AdoBridgeLoop -DryRun # sanity check, mutates nothing # daily use — two terminals: Run-AdoBridgeLoop # A: the producer Run-AgentFactoryLoop # B: the consumer # (or run AgentFactory's dispatcher as the consumer: # one fresh headless session per task)
    🎚️Process templates are config. Agile (New → Active → Resolved → Closed) out of the box; Scrum or custom flows are a few state names in ado-bridge.config.json. requiredTag makes import opt-in per item (e.g. #ai-agent).

    4 · Any source can be a producer

    ADO is the first adapter, not the design. A producer is any agent that can honour the contract:

    1.Stable dedup key in the task title (AB#123:, GH#456:, JIRA-789: …) so passes are idempotent.
    2.A design pass — raw items become implementable briefs (spec + acceptance criteria) before they're queued; the producer is an agent, so this is where it earns its keep.
    3.Queued task via the HTTP API — the board stays the single human checkpoint regardless of where work originates.
    4.Status reflected back — review and done flow to the source (comment + close in ADO today). The natural next outbound step: opening the PR from the pushed branch automatically — submit guardrails already guarantee the branch is on origin.
